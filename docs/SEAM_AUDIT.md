# SEAM AUDIT — `rustc_codegen_clr` vs `rustc_codegen_ssa`

## 1. Executive summary

This audit covers 16 codegen surfaces where `rustc_codegen_clr` reimplements the `rustc_codegen_ssa` lowering step. Postures split as **2 delegate** (`caller_location`, `vtable_dyn`), **9 fork** (`entry_fn`, `call_terminator`, `fn_abi`, `intrinsics`, `switchint`, `rvalue_cast`, `statics_const`, `unwind_cleanup`, plus the call-terminator block layer), and **5 mixed** (`assert_terminator`, `drop`, `discriminant`, `TLS`, `statics_const`/`assert` overlap). Forking is mostly principled: the backend substitutes a calling-convention-agnostic, by-value CIL convention for the entire `PassMode`/`FnAbi` machinery (a self-consistent closed world, not drift), and re-derives a `.NET` try/catch region model for unwinding that has no LLVM analogue. There are **10 CONFIRMED real gaps** reachable from safe stable Rust on `x86_64-unknown-dotnet`: **4 LOUD** (clean ICE — I3-acceptable) and **6 SILENT_WRONG** (observable miscompile — the dangerous class). The 6 silent-wrong gaps cluster into exactly two structural roots: (a) the `UnwindAction::Terminate` edge being dropped to `None` instead of aborting — **4 of the 6 silent-wrong gaps are the same bug surfaced on different terminators/reasons**, and (b) two independent value-correctness omissions (overflow-check elision, nonzero-offset `Indirect` const). **Highest-leverage structural recommendation: converge `handler_from_action`'s `Terminate(_)` arm (`src/basic_block.rs:103-104`) onto the already-correct `UnwindTerminate`-terminator `FailFast` path (`src/terminator/mod.rs:679`), and dispatch on the reason via the real `PanicCannotUnwind`/`PanicInCleanup` lang items — one fix closes 4 of the 6 silent-wrong gaps at once.**

## 3. Per-surface findings

### entry_fn — posture: forks (sound)
Reference: `base.rs:501-580 create_entry_fn` / `maybe_create_entry_wrapper`. Backend hand-rolls the shim in `src/lib.rs:199-257` + `cilly/src/entrypoint.rs`. **Clean for reachable cases.** Main, `Result`/`ExitCode`-returning main (`lang_start`), and `#![no_main]` are all correct (P2-S3). The two divergences — `argc`/`argv` not forwarded into `lang_start`, and the missing `MultipleMainFunctions` diagnostic — are both unobservable on `.NET` (args come from `Environment.GetCommandLineArgs`; there is no C `main` symbol namespace to collide in). UEFI/`#[start]` branches are dead for this target. No confirmed gap.

### caller_location — posture: delegates
Reference: `block.rs:1884-1893` + `mod.rs:592-616`, both delegating the scope-walk to `Body::caller_location_span`. **Clean.** The backend delegates the hard inlining-scope walk to the identical shared helper and only hand-rolls the own-frame `&Location` CIL index, which is provably correct (`fn_abi.args.len()==arg_count+1` ⇒ `LdArg(arg_count)` hits the trailing param). The sole residual — the untupled `rust-call` closure ABI ignoring `num_untupled` — requires the unstable `closure_track_caller` feature, so it is not reachable from safe stable Rust. No confirmed gap.

### call_terminator — posture: forks (deliberate, target-correct)
Reference: `block.rs:905 codegen_call_terminator`. The backend never consults `PassMode`; it passes/returns everything by its by-value Rust type as a CIL value, which is sound because CIL is calling-convention-agnostic and the same sig drives caller and callee. **Residual gaps:** #3 (`UnwindAction::Terminate` on a call edge → no abort, med). The other three "gaps" in the raw audit are non-reachable: variadic-fn-ptr call (unsafe-only, untested), `TailCall`/`become` (`todo!`, unstable `explicit_tail_calls`), and the `compiler_builtins`-upstream-monomorphization guard (not applicable to this link model).

### fn_abi — posture: forks (sound)
Reference: `callconv/mod.rs:40-75 PassMode`. **Clean — no reachable miscompile.** The backend deliberately forks the entire ABI to a uniform by-value CIL convention; `Ignore`/`Direct`/`Pair`/`Cast`/`Indirect` are all re-modeled self-consistently (ZSTs kept as `Void` params to preserve `LdArg(n)` indices). `PassMode` only describes a real on-the-wire contract at the native/FFI boundary, which on this target is serviced by symbol-keyed linker builtins and the `call_managed` interop path — neither depends on this sig's `PassMode` fidelity. Flagged for a *separate* follow-up (not confirmed here): the `rust_call` tuple-splitting fork and the `c_variadic` tail both read structure and warrant their own audit.

### intrinsics — posture: forks
Reference: `intrinsic.rs:57 codegen_intrinsic_call`; `must_be_overridden = !has_body`. The safety net is asymmetric: a missing arm for a *body-having* intrinsic falls back to a real call, but a missing arm for a *bodyless* intrinsic `span_bug!`s. **Residual gaps:** #7 signed `atomic_max`/`atomic_min` (ICE, med, stable-reachable via `AtomicI32::fetch_max/min`) and #8 `atomic_singlethreadfence` (ICE, med, stable-reachable via `compiler_fence`). The remaining bodyless gaps (`nontemporal_store`, `unaligned_volatile_store`, the three `volatile_copy`/`set` forms, the SIMD `gather`/`scatter` wall) are real ICE-on-fallthrough but only reachable via nightly `core_intrinsics` — no std/core/alloc caller. No silent-wrong here: the ordering approximations (atomic load/store/fence ignore the const `ORD`) are deliberate `.NET`-memory-model decisions, benign on CoreCLR's strong model.

### assert_terminator — posture: mixed
Reference: `block.rs:706-805 codegen_assert_terminator`. The variant→lang-item→operand mapping is **fully and correctly covered** (all 12 coroutine + all overflow/div/rem lang items, `BoundsCheck`/`MisalignedPointerDereference`/`InvalidEnumConstruction` special-cased exactly as the reference). **The one divergence is gap #1:** the backend never consults `sess().overflow_checks()` and never const-folds the assert `cond` (`builder.rs:357 const_to_opt_u128` is a dead `todo!()` stub), so an optional overflow assert inlined from a `#[rustc_inherit_overflow_checks]` helper into a release crate becomes a live panic where native rustc silently wraps — a latent panic-vs-wrap miscompile.

### switchint — posture: forks
Reference: `block.rs:391-527 codegen_switchint_terminator`. **Clean — no correctness gaps.** The backend lowers `SwitchInt` to a faithful value-ordered equality cascade, semantically equivalent for every legal input. Discriminant coverage is complete (`Int`/`Uint`/`Bool`/`Char`; the `_ => todo!` arm is dead under the MIR invariant), 128-bit discriminants route to dedicated `eq_i128`/`eq_u128` builtins with no truncation, and signed/negative discriminants round-trip correctly. Everything the reference does that the backend omits (const fast-path, FastISel two-target shapes, branch weights, funclet resolution) is pure optimization or LLVM-ISA-specific — RyuJIT owns jump-table lowering one layer down.

### drop — posture: mixed
Reference: `block.rs:607-704 codegen_drop_terminator`. The backend delegates glue selection and drop-flag elaboration to rustc, and wires the cleanup edge structurally via `term.unwind()`; it forks only the terminator lowering (empty/monomorphic/dynamic), faithfully mirroring the reference incl. the dyn null-check and slot-0 drop-glue assumption. **Residuals:** gap #4 (`UnwindAction::Terminate` on Drop → `None`, low — the shared root of the unwind-action bug), and the missing defensive `drop.is_none()` assert on un-expanded async-drop suspends (not reachable from stable; recommend mirroring the reference's assert to convert a latent silent-wrong into a loud ICE per invariant I3).

### discriminant — posture: mixed
Reference: `operand.rs:433 codegen_get_discr` / `place.rs:256 codegen_set_discr`. The backend hand-rolls the full tag/niche decode-and-encode but tracks the reference structure faithfully and covers every reachable case: direct tags, single-variant, niche untagged/`relative_max==0`/general, pointer-typed niche tags (`Option<&T>`/`Option<Box<T>>` via `int_to_int` Ptr arms), and 128-bit tags via `System.[U]Int128` operators. **No reachable high-impact gap.** The one flagged fork — `set_discr` not masking the niche tag to the field width (reference masks; backend relies on `int_to_int` truncation) — agrees for all power-of-two-width tags, so it is a low-impact latent fork, not a demonstrated miscompile. The `Void`-tag `todo!` in the Direct arm is dead for well-formed `Multiple` layouts.

### rvalue_cast — posture: forks
Reference: `rvalue.rs:414-528` cast match. Most variants are covered (`ReifyFnPointer` goes *beyond* the reference with adapter thunks; `FloatToInt` is correctly saturating with NaN→0; `Unsize` incl. trait-upcasting via `supertrait_vtable_slot`). `dyn*`/`DynStar` does not exist in this nightly — not a gap. **Residual gaps:** #9 narrowing/non-64 `PointerExposeProvenance` (`ptr as u32` → `todo!` ICE, low, stable-reachable; trivial fix — route any int target through cast-to-usize + int_to_int like the existing U64/I64 arm), and a low-impact UB-path silent-wrong on mismatched-size `Transmute` (reference traps via `unreachable_nonterminator`; backend reads/writes garbage — both UB-only paths).

### statics_const — posture: forks
Reference: `consts.rs` (`const_alloc_to_llvm`, `codegen_static_item`) + `common.rs:271 scalar_to_backend`. Coverage is broad: all `GlobalAlloc` variants incl. the dangerous ones (recursive/self-referential statics, nested-anonymous statics — the chrono fix, fn-ptr relocs, vtables, `TypeId`) are explicitly handled. **Residual gaps:** #2 (`ConstValue::Indirect{offset≠0}` discards the offset and reads the wrong sub-object — silent_wrong, med; the sibling `load_scalar_ptr` *does* apply the offset, proving this is an omission not an invariant) and #10 (`#[link_section]` ≠ `.init_array` → `panic!`, loud, not reachable on a portable CIL target). The not-applicable cases (`alloc.mutability`, `#[used]`, partial-uninit chunks, empty-slice sentinel) are correctly benign on `.NET`.

### unwind_cleanup — posture: forks (necessarily)
Reference: `block.rs` resume/terminate/`do_call` unwind dispatch + `landing_pad_for`. The whole LLVM funclet/personality model has no CIL analogue; the backend re-derives a `.NET` try/catch region model (`resolve_exception_handlers` GCs and duplicates cleanup blocks per protected region — the "cleanup duplicated per handler" design). `Resume`/`Cleanup`/`Continue`/`Unreachable` and the global `NO_UNWIND` strip are all covered. **Residual gaps:** #5 (`Terminate`-as-action on Call/Drop/Assert → `None`, the highest-value of this cluster: it is the *same* bug class the P2/B `UnwindTerminate`-terminator fix closed, but only for the terminator form) and #6 (`UnwindTerminate` reason ignored — one generic `FailFast` for both `Abi` and `InCleanup`, wrong message + skips the std panic hook/backtrace). A third, non-reachable residual: a top-level `ReThrow` outside a catch would emit invalid IL, but the region-structuring pass provably prevents it.

### vtable_dyn — posture: delegates
Reference: `meth.rs` + `block.rs:1244-1289`/`640-689`; canonical layout `ty/vtable.rs`. **Clean — faithful delegation.** The backend asks rustc for the vtable via `tcx.vtable_allocation` and materializes its bytes+provenance generically (vtables are `GlobalAlloc::Memory`, so the stubbed `GlobalAlloc::VTable` arm is off the hot path); it only re-derives slot offsets, which match `COMMON_VTABLE_ENTRIES` (asserted). Trait upcasting via `supertrait_vtable_slot`. `dyn*` and VFE/`type_checked_load` are not applicable on this rustc/target. The two divergences (missing `DispatchFromDyn` newtype-peel; i32 vtable-index cap) are both unreachable — every real std smart pointer has its non-1-ZST field at offset 0, and >2³¹ methods is impossible. (Note an *unrelated* known silent-wrong in `unsize.rs:96-108` — `target_size` computed from `source` — out of scope for this surface.)

### TLS — posture: mixed
Reference: `rvalue.rs:662-692 mir::Rvalue::ThreadLocalRef`. The backend correctly delegates the definition side (`THREAD_LOCAL` attr → `[ThreadStatic]`/`_Thread_local`) and covers `ThreadLocalRef` case 2 (direct local thread-local), with a verified per-thread `.tcctor` init on main and std-spawned threads. **Residual gap:** the cross-crate thread-local shim (`ThreadLocalRef` case 1) is a `todo!` (loud), but it needs unstable `#![feature(thread_local)]` *and* a cross-crate access — std's `thread_local!` routes through the runtime-key PAL arm, not the native path, so this cannot arise from safe stable single-crate Rust. Orthogonal documented residuals (not codegen-step gaps): thread-local Drop/destructors don't run on thread exit (leak-on-exit), and a `#[thread_local]` accessed on a raw managed thread sees zeroed storage.

## 4. Suggested follow-up slices

Ordered by leverage. Each is scoped like a P2-S3 slice: one focused fix + a differential regression crate + keep the I1 fatal type-gate green.

### Slice A — Converge the `UnwindAction::Terminate` abort path (closes gaps #3, #4, #5, #6)
Highest leverage: **four of the six silent-wrong gaps are one structural bug.** Replace `handler_from_action`'s `UnwindAction::Terminate(_reason) => None` (`src/basic_block.rs:103-104`) so the edge routes to the existing `FailFast` machinery used by the `UnwindTerminate` *terminator* (`src/terminator/mod.rs:679`), and dispatch on `reason` via the real `PanicCannotUnwind`/`PanicInCleanup` lang items so the message + backtrace + panic-hook behavior match native (fixes #6 simultaneously). Regression crate: the double-panic-in-derived-`Clone` repro (array/tuple field → `Terminate(InCleanup)`) **plus** an `extern "C"` fn owning a `Drop` value that panics (→ `Terminate(Abi)`); assert the process aborts uncatchably (`catch_unwind` does *not* absorb it) rather than printing `"REACHED"`. This converts a catchable, cleanup-running exception back into Rust's guaranteed hard abort.

### Slice B — Overflow-check elision for inlined inherit-overflow asserts (closes gap #1)
Add the `if !sess().overflow_checks() && msg.is_optional_overflow_check()` gate to the Assert arm (`src/terminator/mod.rs:492-554`), forcing `const_cond = Some(expected)` and eliding the panic block so the value wraps — mirroring `block.rs:723-731`. The assert `cond` is a runtime overflow flag, so the CIL optimizer cannot recover this; it must be done at lowering. Regression crate: a `--release` crate calling a cross-crate `#[rustc_inherit_overflow_checks]` helper (e.g. `<u8 as Add>::add(x, 200)`) with an overflowing `black_box` input; assert the `.NET` output equals native release (wrapped `44`), not a panic.

### Slice C — Apply the `Indirect` const offset (closes gap #2)
In `create_const_from_data` (`src/operand/constant.rs`), stop discarding `offset_bytes` (`let _ = offset_bytes`): apply it in both the scalar fast-path and the by-ref `add_allocation` path, exactly as the sibling `load_scalar_ptr` already does and as the reference does via `const_ptr_byte_offset`. Regression crate: a `--release` build that triggers GVN to const-propagate a mid-array sub-object (`ARR[2]` out of a 4-element const array via an `#[inline(never)]` sink); assert it prints `cccc`, not `aaaa`. This is a structurally-valid-CIL-but-wrong-bytes miscompile that I1 cannot catch, so a runtime differential test is the only guard.

### Slice D — Enumerate the missing bodyless intrinsic + cast arms (closes gaps #7, #8, #9)
Three small loud-ICE fixes, batchable. (#7) Add `"atomic_max" | "atomic_min"` arms in `src/terminator/intrinsics/mod.rs` wired to the existing signed-aware `atomic_max`/`atomic_min` helpers (`utilis.rs:283/320`), beside the existing `atomic_umin`/`atomic_umax`. (#8) Extend the `"atomic_fence"` arm (`mod.rs:262`) to also match `"atomic_singlethreadfence"` (a compiler-only fence → `Thread.MemoryBarrier()` or no-op). (#9) Route any integer target in the `PointerExposeProvenance` arm (`src/rvalue.rs:218`) through cast-to-usize + `int_to_int`, like the existing U64/I64 arm. Regression crate: `AtomicI32::fetch_max/fetch_min`, `compiler_fence(SeqCst)`, and `ptr as u32`/`as u16`/`as i8` in one safe-stable program; assert it compiles and matches native (currently each ICEs at codegen). Gap #10 (`#[link_section]` ≠ `.init_array`) is intentionally **left as a loud wall** — no `.NET` section-table equivalent — so it is excluded from this slice.
